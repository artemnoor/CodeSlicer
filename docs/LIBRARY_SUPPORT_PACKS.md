# Library Support Packs Reference

Support packs map framework-specific patterns into standard semantic edges of the impact graph.

## Pattern Matchers and Edge Emitters

Each support pack json contains `edge_rules` matching:
- Decorators (e.g. FastAPI `@app.post("/orders")`)
- Call Receivers/Methods (e.g. `client.post`)
- Import paths
- Constructor parameters type annotations

Matched items emit:
- `ROUTE_HANDLES` (linking route to entrypoint handler method)
- `CALLS` (linking callers to dependencies)
- `TESTS` (linking test calls to route URLs)
- `DEPENDS_ON` (linking components to package frameworks)

## Tested Patterns & Coverage Limits

> [!WARNING]
> Support packs are NOT production-complete for arbitrary codebase patterns. They are verified and tested ONLY for the specific patterns documented below.

### 1. FastAPI Support Pack
- **Tested & Covered Patterns**:
  - Direct HTTP decorators on `FastAPI` application instance (e.g. `@app.get(...)`, `@app.post(...)`).
  - Route handlers using `APIRouter` (e.g. `@router.post(...)`, `@router.get(...)`).
  - Prefix routing composition using `app.include_router(router, prefix="/api")` and nested `APIRouter(prefix="/orders")` declarations.
  - Parameter dependency injection via `Depends(...)` (e.g. `service: OrderService = Depends(get_order_service)`), resolving the edge from the handler to the provider function.
- **Unsupported Cases**:
  - Dynamic route generation at runtime.
  - Prefix parameters (e.g. `/api/{version}/items`).
  - Advanced router factories or class-based route decorators.

### 2. React Support Pack
- **Tested & Covered Patterns**:
  - React Component function declarations (starting with uppercase).
  - Custom React Hooks (starting with `use` followed by uppercase).
  - Component-to-Hook calling relationships (e.g. `OrderForm` calling `useOrders`).
  - Hook-to-API-client-method calling relationships (e.g. `useOrders` calling `postOrder`).
  - API-client-method to HTTP endpoint literal mapping (e.g. `postOrder` calling `fetch('/api/orders')` or `axios.post('/api/orders')`).
  - Frontend-to-backend endpoint matching, matching *only* if the literal route endpoint string matches a known route node in the graph.
- **Unsupported Cases**:
  - Class-based React components.
  - Dynamic endpoint URLs built using variables or string concats (e.g. `fetch('/api/' + resourceId)`).
  - Framework routing packages (React Router, Next.js routing) unless mapped to literal static routes.

### 3. Dependency Injector (`dependency-injector`) Support Pack
- **Tested & Covered Patterns**:
  - Declarative Container definitions subclassing `containers.DeclarativeContainer`.
  - Singleton providers (e.g. `providers.Singleton(OrderRepository)`).
  - Factory providers with constructor bindings (e.g. `providers.Factory(OrderService, repository=order_repository)`).
  - Dynamic container provider calls (e.g. `container.order_service()`).
  - Inferred service-to-repository dependencies mapped dynamically with complete evidence.
- **Unsupported Cases**:
  - Resource providers and configuration providers.
  - Dynamic container configurations loaded from external YAML/JSON files.
  - Positional argument constructor injection inside container providers.

## Installation Validation Gates

1. Candidate support pack is generated by an AI research assistant.
2. It must pass strict json schema audit checks (contains library, version_range, valid edge rule formats with `emit.to` and `emit.kind`).
3. It must pass local playground validation tests showing zero resolver rule engine errors.
4. Once verified, it is installed in the local store registry.
